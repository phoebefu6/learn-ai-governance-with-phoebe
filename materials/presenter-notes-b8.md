# Presenter notes - Practitioner session 8: The AI governance operating model

**Session goal (one line).** Practitioners can assemble the parts of the course into one running operating model - a system registry at the centre, one owner, and AI/data/privacy handled as one system.

## Run of show (45 min)
- **0-3 - Welcome.** From parts to a running system: this session wires b1-b7 into an operating model that keeps working.
- **3-22 - Operating model (Part 1).** The AI system registry at the centre - every system, its tier, owner, obligations, and status. If it is not in the registry, it is not governed.
- **22-40 - One system (Part 2 + build-along).** AI governance, data governance, and privacy run off one data map and one owner; then the build-along "Score the model, plan 90 days."
- **40-45 - Close.** Wrap the practitioner track and point back to the leader-track program view (a6).

## Preflight
- Open b8, "Expand all", confirm the operating-model and one-system SVGs render.
- Have the b1-b7 outputs (classifications, checklists, FRIA/conformity maps) available so the operating model is built from real artifacts.
- Be ready to name the stricter-of-frameworks principle before presenting it.

## Never-cut beats
- **The registry is the backbone.** Classification, obligations, risk method, management system, evals, and monitoring all hang off it. No registry entry means no governance.
- **Run FRIA and DPIA together.** The Act's FRIA and the GDPR/PDPA DPIA overlap on the same question - who is affected and how do we protect them - so run them off one data map and one owner, not two disconnected forms.
- **Stricter of the frameworks.** Build to the EU AI Act floor, add ISO for proof, and where frameworks overlap take whichever demands more; one conservative standard beats several partial ones.

## Cuts if running long
- Score the model as a quick self-rating and draft only the first 90-day move.
- Summarise the FRIA/DPIA integration verbally.

## Quiz answers
1. **B** - An AI system registry / inventory - every system, its tier, owner, obligations, and status - sits at the centre.
2. **C** - Run the FRIA and the DPIA together, off one data map and one owner, since they overlap.
3. **A** - Build to the EU AI Act floor, add ISO for proof, and take whichever framework demands more where they overlap.

## Common questions + crisp answers
- "Do we need separate AI, data, and privacy teams?" No - one operating system off one data map and, ideally, one owner; silos are the failure mode.
- "FRIA or DPIA - which do we run?" Both, but together; they answer the same underlying question.
- "Which framework wins when they conflict?" The stricter one - build to it once rather than maintaining several partial standards.

## Teaching tips
- This is the assembly session - spend it wiring b1-b7 outputs into the registry, not re-teaching them. The registry is the deliverable.
- Make "not in the registry means not governed" the memorable rule; it is the single test for whether the operating model is real.
- Run FRIA and DPIA together on one example so learners feel the overlap rather than hearing about it - one data map, one owner.
- Push the stricter-of-frameworks principle as a simplification, not extra burden: build to one conservative standard instead of maintaining several partial ones.
- Close the practitioner track by connecting back to a6's program view - the leaders own the program, the practitioners built its parts.

## Bridge / close
This ends the practitioner track. Point learners to the a6 program hub and 90-day plan so the parts they built get an owner and a cadence.

## Compliance caution
Educational, not legal advice; the high-risk regime is under live amendment (the proposed Digital Omnibus) and the ISO/IEC 42001 control set should be verified against the standard - confirm your operating model against the Regulation and get advice.
