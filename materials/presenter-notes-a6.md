# Presenter notes - Leader session 6: Building an AI governance program

**Session goal (one line).** Leaders leave with a program hub in mind - one owner, connected to data governance and evals - and a 90-day plan to stand it up.

## Run of show (45 min)
- **0-3 - Welcome.** Close the loop: five sessions of pieces become one running program today.
- **3-20 - Concepts (Part 1 + Part 2).**
  - Part 1 (approx 9 min): the program hub - inventory, classification, obligations, owner. The number-one failure is scattered ownership.
  - Part 2 (approx 8 min): the operating-system view - AI governance, data governance, and model evaluation are one system, not three silos.
- **20-40 - Exercises.**
  - Exercise 1 "Score your program artifacts" (approx 12 min): rate what exists vs what is missing.
  - Exercise 2 "Write a 90-day plan" (approx 8 min): first three moves with an owner and a date.
- **40-45 - Q&A.** Send them off with the leader-track wrap and the handoff to the practitioner track.

## Preflight
- Open a6, "Expand all", confirm the program-hub SVG and the operating-system diagram render.
- Have the a1-a5 cheat sheets available so the 90-day plan pulls from real prior work.
- Be ready to name a plausible AI governance lead role for the Vantage case.

## Never-cut beats
- **One named owner.** A program with nobody holding the whole is the classic failure mode; legal holds the law, engineering holds the models, and the gap in the middle is where governance dies.
- **One operating system, not three silos.** Models depend on governed data; evals produce the evidence a regulator or customer wants. Wire them together.
- **Compliance is a habit, not a project.** Tie a review cadence to new systems and features so the program stays alive between audits.

## Cuts if running long
- Score artifacts as a quick self-rating rather than a full walk-through.
- Have learners draft only the first of the three 90-day moves in the room.

## Quiz answers
1. **B** - Scattered ownership: nobody owns the whole. A program needs one named AI governance lead.
2. **C** - Because models depend on governed data, and evals produce the evidence a regulator or customer wants.
3. **A** - Tie a review cadence to new systems and features so the program stays alive; compliance is a habit, not a one-off.

## Common questions + crisp answers
- "Who should own AI governance - legal or engineering?" A dedicated lead who connects both, with executive backing; not either silo alone.
- "Where do we actually start in 90 days?" Inventory and classify first, name an owner, then stand up one review cadence.
- "How does this connect to our existing data governance?" It reads from it - same data map, ideally same owner; do not build a parallel stack.

## Teaching tips
- This is the payoff session - make the 90-day plan concrete enough that someone could start Monday. Owner, first move, date.
- Push hard on ownership. The most common quiet failure is a program everyone supports and nobody owns; name a role, not a committee.
- Draw the operating-system view as one diagram, not three boxes; the whole point is that AI, data, and evals are wired together.
- "Compliance is a habit, not a project" is the line to send them out on. A program that only wakes for an audit has already failed.
- Pull the plan from real prior work (a1-a5 cheat sheets) so it is their program, not a generic template.
